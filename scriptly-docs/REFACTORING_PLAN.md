# 🚀 Scriptly 프로젝트 리팩토링 및 인수인계 계획

본 계획서는 Scriptly 프로젝트의 코드 품질을 높이고, 사용자가 직접 필사하며 구조를 학습할 수 있도록 코드를 재구성하기 위한 단계별 로드맵입니다.

## 🎯 목표
1. **클린 아키텍처 적용**: 계층 간 의존성을 명확히 하고 유지보수성을 향상시킵니다.
2. **필사(Transcription) 최적화**: 코드의 가독성을 극대화하고, 설계 의도가 담긴 주석을 통해 학습 효과를 높입니다.
3. **표준화**: Next.js 15와 FastAPI(Python 3.12)의 최신 관례(Convention)를 엄격히 준수합니다.
4. **지식 전달**: 프로젝트 전반을 이해할 수 있는 상세한 인수인계서를 제공합니다.

---

## 📅 단계별 테스크 리스트

### Phase 1: 기반 다지기 및 도큐멘테이션
- [ ] `scriptly-docs/REFACTORING_PLAN.md` 작성 (완료)
- [ ] `scriptly-docs/HANDOVER_GUIDE.md` 초안 작성 (전체 구조도 포함)

### Phase 2: Backend (scriptly-api) 리팩토링
- [ ] **루트 디렉토리 정리**: `check_db.py`, `migrate_*.py` 등 파편화된 스크립트를 `scripts/` 폴더로 이동 및 체계화.
- [ ] **Domain 계층 강화**: `app/domain` 내 엔티티와 스키마를 순수 비즈니스 로직 중심으로 재정의.
- [ ] **Infrastructure 계층 분리**: DB 세션 관리 및 외부 API(Naver) 통신 로직을 명확히 격리.
- [ ] **Application(Service) 계층 정교화**: AI 오케스트레이션 로직의 가독성 개선 및 필사용 가이드 주석 추가.
- [ ] **Interface(API) 계층 표준화**: FastAPI 라우터 구조를 도메인별로 일관성 있게 재구성.

### Phase 3: Frontend (scriptly-web) 리팩토링
- [ ] **컴포넌트 구조 재설계**: `src/components`를 `features/`(기능별), `shared/`(공통), `layout/`으로 재배치.
- [ ] **Server/Client 컴포넌트 최적화**: `use client` 선언 최소화 및 데이터 페칭 전략 최적화.
- [ ] **상태 관리 및 Hooks 정리**: 비즈니스 로직을 커스텀 훅으로 추출하여 컴포넌트 가독성 향상.
- [ ] **타입 시스템 정비**: 전역 타입 정의 및 API 응답 타입의 엄격한 준수.
- [ ] **필사 포인트 추가**: 핵심 컴포넌트에 코드 흐름과 설계 이유를 설명하는 주석 삽입.

### Phase 4: 검증 및 마무리
- [ ] 리팩토링 후 전체 기능 테스트 (Curation, Archive, Insight Lab).
- [ ] `HANDOVER_GUIDE.md` 최종 업데이트 (실행 방법, 트러블슈팅 가이드 포함).
- [ ] 최종 결과 보고 및 인수인계 완료.

## 🛠 리팩토링 원칙
- **변수 및 함수명**: 직관적이고 서술적인 이름을 사용하여 주석 없이도 의도가 드러나게 함.
- **코드 일관성**: 동일한 목적의 로직은 프로젝트 전체에서 동일한 패턴을 따름.
- **학습 친화적**: 복잡한 인라인 로직보다는 단계별로 명확히 분리된 함수형 스타일 지향.
